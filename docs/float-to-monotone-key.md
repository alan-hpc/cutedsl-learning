# `float_to_monotone_key` 原理详解

> 对应代码：[`topk/select_radix_cute.py`](../topk/select_radix_cute.py)

`float_to_monotone_key` 把一个 IEEE754 `float32` 转成 `uint32`，转换后**按无符号整数比较的顺序，和原始浮点数按数值比较的顺序完全一致**。这是 radix select / radix sort 能够处理浮点数据的前提——radix 算法本身只会逐字节比较无符号整数，不认识符号位、指数、尾数这些浮点语义,所以必须先把 float "翻译" 成一个顺序等价的整数 key。

```python
@cute.jit
def float_to_monotone_key(value: Float32) -> UInt32:
    bits = cute.bitcast(value, UInt32)
    sign_bit = bits >> 31                      # 0 = 正数, 1 = 负数
    flip_mask = (Uint32(0) - sign_bit) | Uint32(0x80000000)
    return bits ^ flip_mask
```

---

## 1. 为什么需要这一步转换

文件头部注释描述的算法是 **Radix-Select + Gather**：

- 阶段 A：对 32-bit key 做 4 轮、每轮 8-bit 的 radix 扫描（256 桶 histogram），从最高有效字节开始逐位收紧范围，最终锁定"第 K 大"对应的阈值 key。
- 阶段 B：用这个阈值做一次 gather，把所有 `key >= 阈值` 的元素紧凑写出。

整个过程中，radix 算法只关心 **无符号整数的字典序**：从最高字节开始比较，哪个字节大,对应的数就大。如果直接把 `float32` 的原始 bit 模式当 `uint32` 塞进这套流程，会在两个地方出错（下面第 3、4 节会证明）：

1. 负数的符号位是 1，正数是 0——当成无符号数看，**所有负数都比所有正数"大"**，顺序整体是反的。
2. 在负数内部，bit 模式越大，对应的数值反而越小（绝对值越大,数值越负）——**负数内部的顺序也是反的**。

`float_to_monotone_key` 要做的，就是用一次 XOR，把这两处"反"全部纠正过来，使结果在保持"逐字节比较"的可行性下，整数序 = 浮点数值序。

## 2. IEEE754 float32 的位布局

```
位:   31        30 ... 23        22 ... 0
内容: [ 符号 S ][   指数 E (8位)  ][  尾数 M (23位) ]
```

对归一化数（`E` 在 1~254 之间），数值为：

$$\text{value} = (-1)^S \times \Big(1 + \frac{M}{2^{23}}\Big) \times 2^{\,E-127}$$

把符号位拿掉之后剩下的 31 位，记为：

$$I = E \cdot 2^{23} + M$$

这 31 位本身就是一个普通的无符号整数（指数占高位，尾数占低位）。下面证明的核心结论是：

> **对同符号的浮点数，$I$ 越大，数值的绝对值就越大。** 即"剩余 31 位当整数读"天然按绝对值排好序了。

在证明这一点之前，先把 float32 的编码规则完整展开——单调性证明（第 3 节）和后面的取值范围、精度讨论都建立在这套规则之上。

### 2.1 五种位模式，对应五类数值

`(E, M)` 并不是任何组合都按 3.1 节的归一化公式解释。32 位被拆成 5 类，由 `E` 和 `M` 是否为边界值共同决定：

| `E`（8 位） | `M`（23 位） | 类别 | 数值公式 |
|---|---|---|---|
| `0` | `0` | ±0 | $(-1)^S \times 0$ |
| `0` | `≠0` | 次正规数（denormal / subnormal） | $(-1)^S \times \dfrac{M}{2^{23}} \times 2^{-126}$ |
| `1 ~ 254` | 任意 | 正规数（normalized） | $(-1)^S \times \Big(1+\dfrac{M}{2^{23}}\Big) \times 2^{\,E-127}$ |
| `255` | `0` | ±∞ | — |
| `255` | `≠0` | NaN | — |

之所以要单独分出"次正规数"这一类，是因为正规数公式里有一个隐含的 **前导 1**（即 $1 + M/2^{23}$ 里固定的那个 `1`）——这个 1 不占用任何存储位，是 IEEE754 的一个小技巧：既然正规数永远写成 `1.xxxxx × 2^E` 的科学计数法形式，这个最高位必然是 1，干脆不存，省出一位给尾数用，相当于免费多得一位精度。

但这个技巧只在 `E ≥ 1` 时成立。当指数已经降到最低（`E=0`）、还要表示比"最小正规数"更小的数时，没有更小的指数可用了，只能放弃"隐含前导 1"，让尾数退化成纯小数 $M/2^{23}$（不加 1），同时把指数固定用 `2^{-126}`（而不是 `2^{-127}`，这样设计是为了让次正规数和正规数在 `E=0/1` 的交界处刚好无缝衔接，下面 2.4 节会验证这一点）。这就是"次正规数"——用损失精度（不再有隐含 1）的代价，换取能表示比最小正规数更接近 0 的值，避免在 0 附近留下一段"表示不了任何数"的空洞。

### 2.2 指数偏置（bias）：为什么是减 127

`E` 是一个 8 位无符号数（0~255），但浮点数的指数既要能表示很大的数也要能表示很小的数,需要正负都有。IEEE754 没有再单独留一个指数符号位,而是约定"偏置（bias）"：真实指数 = 存储的 `E` 减去 127。

- `E=1`（最小正规指数）→ 真实指数 $1-127=-126$
- `E=127` → 真实指数 $0$ → 这一档表示 $1.xxx \times 2^0$，即 1.0 附近的数
- `E=254`（最大正规指数）→ 真实指数 $254-127=127$

`E=0` 和 `E=255` 被保留给次正规数/零和无穷/NaN,不参与"减 127"的常规解释,这也是为什么次正规数公式里直接写死用 `2^{-126}`（对应 `E=1` 的真实指数）,而不是套用 `E=0` 算出的 `2^{-127}`。

### 2.3 取值范围一览

| 量 | 数值 | 对应 bit 模式 |
|---|---|---|
| 最小正次正规数 | $2^{-126} \times 2^{-23} = 2^{-149} \approx 1.4013 \times 10^{-45}$ | `0x00000001` |
| 最大次正规数 | $\dfrac{2^{23}-1}{2^{23}} \times 2^{-126} \approx 1.1755 \times 10^{-38}$ | `0x007FFFFF` |
| 最小正正规数 | $2^{-126} \approx 1.1755 \times 10^{-38}$ | `0x00800000` |
| 1.0 | $1.0$ | `0x3F800000` |
| 最大有限正规数 | $(2-2^{-23}) \times 2^{127} \approx 3.4028 \times 10^{38}$ | `0x7F7FFFFF` |
| +∞ | — | `0x7F800000` |
| NaN（其中一种） | — | `0x7FC00000` |

注意"最大次正规数"和"最小正正规数"几乎相等（只差最后一位精度）——这正是 2.1 节说的"无缝衔接"：次正规数把 $[0,\,2^{-126})$ 这段区间，用线性（而非指数）间隔的方式填满，紧接着正规数从 $2^{-126}$ 开始接管，往上变成指数间隔。

### 2.4 精度与 ULP：为什么间隔会随指数翻倍

固定指数 `E` 时，相邻两个尾数（`M` 和 `M+1`）对应的数值之差是一个常数：

$$\text{ULP}(E) = \frac{1}{2^{23}} \times 2^{\,E-127} = 2^{\,E-150}$$

`ULP` 是 *unit in the last place*，即"最后一位尾数代表的数值间隔"。可以看到 ULP 直接和 `E` 挂钩：**指数每增大 1，相邻可表示数之间的间隔就翻一倍**。这正是第 3 节证明单调性时,"尾数溢出进位、指数加 1"那一步衔接得天衣无缝的根本原因——因为尾数恰好被设计成：在每个指数档位内，用 $2^{23}$ 个等间隔的点，刚好填满到下一档位的间隔（翻倍后的区间，用一半精度的点数也能填满,因为绝对间隔也翻倍了）。

这也解释了 float32 常说的"7 位有效十进制数字"精度从哪来：$2^{-23} \approx 1.19 \times 10^{-7}$，即相对精度恒定在约 $10^{-7}$ 量级，但**绝对精度**（ULP 的实际大小）随数值变大而线性变大——1.0 附近 ULP 约 $1.19\times10^{-7}$，而 $10^{30}$ 附近 ULP 已经粗到 $10^{23}$ 量级。这是"浮点数"和"定点数"最本质的区别：浮点数用指数换来了跨越约 80 个十进制数量级的动态范围,代价是绝对精度不是常数。

### 2.5 舍入（rounding）简述

当一次运算（比如两个 float32 加法）的精确结果落在两个可表示值之间时，硬件需要按某种规则"舍入"到最近的可表示值。IEEE754 默认规则是 **round-to-nearest-even**（最近偶数舍入）：舍入到最接近的可表示值；如果精确结果正好在两个可表示值正中间（这种情况只会发生在二进制下的"五五开"，类似十进制的 *.5 舍入歧义），则选尾数最低位是 0（"偶数"）的那个。这个细节和本文档主线（单调 key 转换）关系不大，但理解它有助于理解"为什么同一个数学表达式换一种计算顺序,结果可能在最后一位上有微小差异"。

### 2.6 和其它常见浮点格式的对比

这个仓库的 `gemm/`、`attention/` 目录大量涉及混合精度 kernel，下表把 float32 和几种 GPU kernel 里常见的格式放在一起对比（格式：符号 + 指数位 + 尾数位）：

| 格式 | 符号 | 指数 | 尾数 | bias | 典型用途 |
|---|---|---|---|---|---|
| float64 | 1 | 11 | 52 | 1023 | 高精度累加 |
| **float32** | **1** | **8** | **23** | **127** | 本文讨论对象，通用计算 |
| bfloat16 | 1 | 8 | 7 | 127 | 训练/推理权重，指数范围和 float32 一致，精度更低 |
| float16（IEEE binary16） | 1 | 5 | 10 | 15 | 精度比 bfloat16 高，但动态范围小很多 |
| tf32（NVIDIA Tensor Core 内部格式） | 1 | 8 | 10 | 127 | 矩阵乘内部累加精度，指数范围同 float32 |
| fp8 e4m3 | 1 | 4 | 3 | 7 | 低精度推理/训练，精度优先 |
| fp8 e5m2 | 1 | 5 | 2 | 15 | 低精度推理/训练，动态范围优先 |

可以看到所有这些格式共享同一套"符号 + 指数 + 尾数 + 偏置"结构，区别只是位宽分配不同——指数位越多,动态范围越大但精度越低；尾数位越多,精度越高但动态范围越小。`bfloat16` 之所以在深度学习里很受欢迎，正是因为它保留了和 float32 完全相同的指数位宽（8 位），只是简单截断尾数——这样从 float32 转 bfloat16 几乎不需要考虑动态范围溢出，只是精度下降，转换逻辑非常简单（只是高 16 位截断）。本文证明的"同号单调性"（第 3 节）对这些格式同样成立，因为它们都遵循同一套"指数在高位、尾数在低位"的编码约定——`float_to_monotone_key` 的思路如果要扩展到 bf16/fp16 的 radix sort，做法是完全一样的，只是位宽和掩码常数要换成对应格式的宽度。

这不是巧合，而是 IEEE754 编码时刻意设计出来的性质（这样设计是为了让硬件比较器、排序网络可以直接复用整数比较逻辑）。

## 3. 单调性证明

### 3.1 同一个指数内，尾数递增 → 数值递增

固定 `E`，让 `M` 从 0 增加到 `0x7FFFFF`：

$$\text{value}(M) = \Big(1 + \frac{M}{2^{23}}\Big) \times 2^{E-127}$$

这是关于 `M` 的线性递增函数（系数 $2^{E-127}/2^{23} > 0$），`M` 每加 1，数值就线性增加一点点。这一段显然单调。

### 3.2 尾数溢出、向指数进位的瞬间，顺序依然接得上

真正需要验证的是"边界衔接"：当 `M` 从最大值 `0x7FFFFF` 再加 1 时，会发生进位——`E` 增加 1，`M` 归零。把整数 $I$ 看成普通二进制数，这一步进位是连续的（`...0_11111111111111111111111` → `...1_00000000000000000000000`，整数值刚好加 1）。问题是：数值层面这一步是否也连续递增、没有跳跃或反转？

取具体的 `E = 127`（即 $2^{E-127}=2^0=1$）来算：

| 状态 | E | M | 数值 |
|---|---|---|---|
| 进位前（最大尾数） | 127 | `0x7FFFFF` | $(1 + \frac{8388607}{8388608}) \times 1 \approx 1.99999988$ |
| 进位后（尾数清零，指数+1） | 128 | `0x000000` | $(1 + 0) \times 2^{1} = 2.0$ |

$1.99999988 < 2.0$ —— **进位后的数值确实更大**,而且二者之间没有其它可表示的浮点数插在中间（这正是"尾数能表示的范围刚好填满一倍区间、不重叠不留缝"的含义：指数每增 1,数值翻一倍,而尾数恰好能把 $[2^{E-127},\,2^{E+1-127})$ 这个区间等分成 $2^{23}$ 份）。

把 3.1 和 3.2 合起来：**整数 $I$ 从 0 一路加到 $2^{31}-1$，对应的浮点数值也严格单调递增，中间没有任何反转或跳跃**。这就是第 2 节结论的完整证明。

### 3.3 小结

正因为这个性质是 IEEE754 编码的设计目标而不是偶然结果，所以"把 float 的位模式按整数读"这件事在很多硬件 / 库的底层实现里都被复用（比如这正是 NVIDIA CUB 的 radix sort 在处理 float key 时采用的同一套思路）——这一函数只是把这个通用技巧搬到了 CuTeDSL 里。

## 4. 加上符号位：从"同号单调"扩展到"全体浮点数单调"

第 3 节证明的是**同符号**内部的单调性。要覆盖正负数混合的情况，还要处理符号位本身。

### 4.1 正数（`sign = 0`）

31 位本身已经按 3.3 的结论排好序了。要做的只是把这一整段数挪到"比所有负数都大"的区间——也就是把最高位（符号位）从 0 改成 1。其余 31 位不动，相对顺序自然保持：

$$\text{key} = \text{bits} \oplus \texttt{0x80000000}$$

### 4.2 负数（`sign = 1`）

负数有两件事需要同时纠正：

1. 符号位要从 1 改成 0（让所有负数整体落到比所有正数都小的区间）。
2. 31 位内部的顺序要整体反转——因为第 3 节证明的是"绝对值越大、$I$ 越大"，但对负数，绝对值越大意味着数值**越小**（更负）。也就是说，对负数，我们需要的关系是"$I$ 越大、数值越小"，这正好和原始 $I$ 的单调方向相反。

**对 31 位做整体按位取反**（`NOT`，等价于 XOR 全 1）可以一次性满足这两点：

- 取反让最高位（原符号位）从 1 变 0 → 满足第 1 点。
- 取反是严格的"减序"变换：若 $I_1 < I_2$，则 $\overline{I_1} > \overline{I_2}$（每一位取反，二进制数的大小关系整体反转）→ 把"$I$ 越大数值越小"的关系，变成"取反后的值越大数值越大" → 满足第 2 点。

于是负数的修正是对全部 32 位取反：

$$\text{key} = \text{bits} \oplus \texttt{0xFFFFFFFF}$$

### 4.3 正负数能不能接得上？（边界：±0）

- `+0.0` 的 bits 是 `0x00000000`，按正数规则：`key = 0x00000000 ^ 0x80000000 = 0x80000000`
- `-0.0` 的 bits 是 `0x80000000`，按负数规则：`key = 0x80000000 ^ 0xFFFFFFFF = 0x7FFFFFFF`

`-0.0` 的 key（`0x7FFFFFFF`）恰好比 `+0.0` 的 key（`0x80000000`）小 1——二者数值上相等，key 只差 1，对 top-K / 排序的最终结果没有实质影响（不会有第三个数插在它们中间）。

### 4.4 NaN / Inf 的注意事项

这个函数没有特殊处理 `NaN`：`NaN` 的 bit 模式（指数全 1、尾数非 0）会被当成"普通数"一样套用上面的规则，得到的 key 会落在 `+Inf` / `-Inf` 附近，但 `NaN` 之间或 `NaN` 与其它数之间本来就没有定义良好的数值序（IEEE754 规定 `NaN` 与任何数比较都返回 false）。如果输入数据可能包含 `NaN`，需要在调用这个函数之前单独过滤或替换,这不是 `float_to_monotone_key` 本身要解决的问题。`+Inf` / `-Inf` 则不受影响,会正确地映射成最大 / 最小的 key（行为和"无穷大的绝对值最大"一致）。

## 5. 从"两条规则"到 branch-free 表达式

把 4.1、4.2 的结论写成真值表：

| `sign_bit` | 需要的 `mask` |
|---|---|
| 0（正数） | `0x80000000` |
| 1（负数） | `0xFFFFFFFF` |

直接写当然可以用 `if/else`，但 GPU 上的分支会带来 warp 内的发散（divergence），所以代码用纯算术/位运算拼出同样的效果：

```python
flip_mask = (Uint32(0) - sign_bit) | Uint32(0x80000000)
```

### 5.1 `0 - sign_bit`：把 0/1 广播成"全 0 或全 1"

`sign_bit` 只能是 0 或 1：

- `sign_bit = 0` → `0 - 0 = 0`（全 0）
- `sign_bit = 1` → `0 - 1`，在 32 位无符号补码运算下回绕成 `0xFFFFFFFF`（全 1）

这是一个常见的"位掩码广播"技巧：把一个 0/1 的布尔结果，扩展成"全 0 或全 1"的整片掩码,用来给后续的位运算当"条件开关"用,完全不需要分支指令。

### 5.2 `| 0x80000000`：补上正数缺的那一位

只看 5.1 的结果还不够：`sign_bit=0` 时算出来是 `0x00000000`,但正数实际需要的 mask 是 `0x80000000`（要去翻转符号位）,不是 0。所以再 `| 0x80000000`：

- `sign_bit=0`：`0x00000000 | 0x80000000 = 0x80000000` ✅ 正数要的 mask
- `sign_bit=1`：`0xFFFFFFFF | 0x80000000 = 0xFFFFFFFF`（最高位本来就是 1，OR 不改变）✅ 负数要的 mask

两种情况都对上了，于是：

```python
key = bits ^ flip_mask
```

就同时实现了 4.1（正数翻符号位）和 4.2（负数全翻转），且全程没有一条分支指令——这也是函数注释里"PTX 一行"的来源：减法、或、异或这几步在编译后通常会被折成极少的整数指令。

## 7. 在 Radix Select 中的角色

回到文件开头的算法描述：每个 block 处理一行数据,要找出"第 K 大"的元素。具体流程依赖这个 key 的两个性质：

1. **可以逐字节（8-bit）做 radix 扫描**：因为 key 是普通 `uint32`,可以直接拆成 4 个字节,从最高字节开始建 256 桶 histogram,每轮根据某个桶的累计计数判断"第 K 大"落在哪个桶里,逐字节收紧候选范围（4 轮刚好覆盖 32 位）。
2. **整数序等价于原始浮点数值序**：这保证了"在 key 空间里做的 radix select"得到的阈值,对应的就是原始浮点数据里真正的第 K 大值——如果直接对原始 float bit pattern 做 radix select,前面第 1 节列的两个反转问题会让结果完全错误（比如所有负数会被排到"更大"的一侧）。

阶段 B 的 gather 则直接复用阶段 A 算出的阈值 key,把 `key >= 阈值` 的原始元素挑出来——这一步也依赖 key 与原始数值严格单调对应,否则"阈值"在 key 空间和在数值空间就不是同一个分界点。

## 8. 小结

| 问题 | 解决方式 |
|---|---|
| 负数 vs 正数,符号位方向反了 | 正数翻符号位、负数全翻转,把负数整体移到比正数小的区间 |
| 负数内部,bit 模式越大数值越小 | 整体取反同时反转了符号位和内部顺序 |
| 不能用分支（GPU warp divergence） | `0 - sign_bit` 把 0/1 广播成全 0/全 1 掩码,再 `\| 0x80000000` 补齐正数情况 |
| Radix select 只认无符号整数序 | 转换后 uint32 的字典序与浮点数值序严格一致,可以直接喂给 8-bit radix 扫描 |
